SDD Pro — De la spécification fonctionnelle à l'application livrée

SDD Pro (Spec-Driven Development) est un framework qui transforme une spécification fonctionnelle écrite en français en une application complète, testée et auditée, exécutée par des agents IA orchestrés sous Claude Code. Version actuelle : v7.0.0-alpha sur la branche next (en cours de stabilisation vers v7.0.0 GA — cf. .claude/docs/scope-reduction-v7-ga.md). La branche main reste sur v6.10.4-LTS (freeze actif jusqu'au 2026-06-18).

Vous décrivez la valeur métier. SDD Pro produit le code, les tests, l'audit qualité et la traçabilité.

Le problème que SDD Pro résout

Le développement logiciel classique perd 30 à 60 % de l'intention initiale entre la rédaction d'une spec et la livraison du code. Causes habituelles :

  • Les agents IA non encadrés dérivent : ajout de features non demandées, refactor non sollicité, choix techniques opaques.
  • La traçabilité spec → code est artisanale et perdue dès la 2ᵉ itération.
  • Les tests, la sécurité, la performance, l'accessibilité sont des étapes optionnelles, souvent négligées.
  • Le Product Owner ne peut pas valider à chaque étape sans lire du code.

SDD Pro répond à ces 4 points avec un pipeline strict, des agents spécialisés et une console de validation humaine.

Comment ça marche en 4 phases

Phase 1 — Cadrage fonctionnel

Une commande unique /feat-generate Auth ouvre un dialogue de cadrage : besoins, acteurs, règles métier, critères d'acceptation. Le résultat est une FEAT structurée (workspace/input/feats/1-Auth.md) avec des identifiants stables (SFD-1, BR-1, AC-1) qui serviront de traçabilité dans toute la chaîne.

Option : /feat-deepen 1 applique 5 techniques d'élicitation (Pre-mortem, Red Team, First Principles, Stakeholder Mapping, Inversion) pour révéler les risques cachés avant de coder.

Phase 2 — Découpe en User Stories

L'agent PO découpe la FEAT en User Stories (1 à 6 max). Chaque US est autonome, validée par le Product Owner, et trace explicitement les SFD/BR/AC/FD qu'elle couvre. Aucun bullet de la spec ne peut être orphelin (validation déterministe via /feat-validate).

Phase 3 — Bootstrap technique

L'agent Architecte bootstrap le projet selon le stack déclaré (28 stacks supportés — .NET / Spring Boot / FastAPI / Node Express côté serveur ; React / Vue / Angular / Blazor côté client ; Shadcn / Vuetify / Radzen Blazor côté UI). Si une base de données est déclarée, il introspecte sa structure et génère les entités correspondantes (Database-First).

Phase 4 — Génération du code

Pour chaque US, deux agents tournent en parallèle :

  • Dev-Backend — endpoints HTTP, services, validation, persistance
  • Dev-Frontend — pages, composants, formulaires, traduction HTML mockup → code design-system

Entre les deux familles, une API Gate automatique vérifie via tests d'intégration HTTP que les endpoints backend honorent leur contrat avant d'autoriser la génération frontend (règle backend-first, depuis 2026-05-07).

Phase 5 — QA & audit qualité

La commande /qa-generate {n} orchestre :

  • Tests unitaires + coverage (xUnit / Vitest / pytest / JUnit / Jasmine / bUnit)
  • Quality scan sonar-like (TODO/FIXME, magic numbers, méthodes longues, debug output)
  • 5 auditeurs spécialisés en parallèle : accessibility (WCAG 2.2 AA), code-review (anti-patterns cross-fichier), security (OWASP Top 10 + STRIDE), performance (Core Web Vitals + SLO API), spec-compliance (vérification AC par AC indépendante)

La console web — votre cockpit de validation

Depuis la v6.10, SDD Pro embarque une console web qui centralise toute la télémétrie projet dans une base SQLite unique (workspace/output/db/console.db). Plus aucun fichier .json ou .jsonl de stats sur le disque — tout est interrogeable, exportable, comparable dans le temps.

cd workspace/console
npm install      # 1ère fois uniquement

Trois pages, trois publics

Dashboard — vue exécutive

  • 4 KPI cards : nombre de FEATs, tests API totaux, issues sécurité, code smells
  • Grille statuts par FEAT : Run / API Gate / Coverage / Quality / Sécurité
  • Audit qualité style SonarQube avec ratings A→E (Vulnerabilities, Code Smells, Coverage). Une ligne par FEAT, cartes uniquement pour les métriques disponibles (pas de placeholder vide).
  • Drill-down : un clic sur une ligne FEAT déplie le détail des issues (fichier, ligne, classe OWASP/CWE, message, règle, fichiers sous-couverts)
  • 4 graphes SVG natifs : coverage par FEAT, quality stacked, tests API passed/failed, donut sécurité

Features — vue métier

  • Vue PO : FEAT → User Stories (pas de plan technique — pour validation Product Owner)
  • Vue technique : FEAT → US → plans back/front (pour Tech Lead / dev)
  • Vue UX : carrousel des mockups HTML par FEAT (rendu avec leur design-system natif via route statique /ui/ — pas de duplication)
  • Validation humaine en un clic : validé / rejeté / en attente — décision tracée (email + timestamp + commentaire optionnel)
  • Live updates : SSE push les modifications du filesystem → l'arbre se met à jour sans rechargement
  • Bouton Rafraîchir dans l'en-tête : re-scan du filesystem (nouveaux .md et .html apparaissent dynamiquement)

Documentation — vous êtes ici

Theme & ergonomie

  • Thème clair / sombre en un clic, persisté en local storage, suit la préférence système au premier chargement
  • Logos adaptatifs (versions claire / sombre selon le thème)
  • Charts modernes avec gradients vifs (indigo / cyan / amber / red / emerald / pink / violet), sparklines sur les KPIs
  • Spinner animé SVG natif (rotation + 3 dots pulse) — pas de GIF, pas de dépendance externe

Les gains concrets

Pour le Product Owner

  • Lecture des US sans toucher au code (Vue PO de la console)
  • Validation à chaque gate (PO → Readiness → Plan → Code) — l'IA attend votre feu vert
  • Reformulation IA des plans techniques en langage métier (bouton intégré console, opt-in via API key Anthropic)
  • Traçabilité 100 % SFD → AC → US → Code (garantie déterministe par /feat-validate)

Pour le Tech Lead

  • Anti-derive strict : aucun fichier hors plan, aucune lib hors stack déclaré, aucun refactor non demandé
  • Build loop borné : 3 itérations max par US, classes d'erreur déterministes ([BUILD_CORRECTIBLE] / [BUILD_BLOCKING])
  • Audit qualité automatique (sécurité, performance, accessibility, code-review, spec-compliance) à chaque livraison
  • ADRs générés automatiquement pour chaque décision structurante

Pour la direction technique

  • Réduction du time-to-market : pipeline 2-3 jours là où le développement manuel prend 2-3 semaines
  • Conformité OWASP Top 10 + WCAG 2.2 AA par défaut, pas en option
  • Télémétrie tokens (coût par agent, par FEAT, par US) pour optimiser le budget IA — table token_usage dans console.db
  • Profiles configurables (~/.sdd/config.team.yml) : un preset strict-prod, un preset dev-only, etc.
  • Serveur MCP optionnel exposant 14 tools SDD Pro à Cursor / Windsurf / Cline / Claude Desktop / scripts CI

Pour qui ?

  • Équipes 2-12 personnes qui veulent industrialiser leur production logicielle
  • Projets en stack .NET / Java / Node / Python côté serveur, React / Vue / Angular / Blazor côté client
  • Contextes réglementés (santé, banque, public) où la traçabilité spec → code est exigée
  • Organisations qui adoptent Claude Code et veulent un cadre opérationnel
