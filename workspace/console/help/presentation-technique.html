SDD Pro — Architecture technique

Cette page documente l'architecture interne de SDD Pro v6.10 : agents, pipeline, base de données, console web, intégrations. Cible : Tech Lead, architecte, développeur framework.

SDD Pro est écrit en Python (stdlib pure pour le moteur, pytest pour les 570 tests). La console web est en Node.js 20+ (Fastify 5 + React 18 via CDN). La télémétrie vit dans une base SQLite WAL. Le code applicatif généré est dans le stack du projet cible (28 stacks supportés).

2 — Les agents

4 agents cœur

AgentModèleRôle
poSonnet 4.6Découpe FEAT en User Stories (min 1, cible 1-3, hard cap 6). Traçabilité SFD/BR/AC/FD.

dev-frontendOpus 4.7Pour UNE US, génère le code client (pages, composants) en traduisant le HTML mockup vers le design-system actif.

Variants strict (v6.2)

  • dev-backend-strict et dev-frontend-strict — Sonnet 4.6, consomment un plan v2 strict-ready (avec inline digest), 3× plus rapides et 5× moins chers que les Opus pour le chemin From-Plan.
  • Opt-in via PlanCacheStrict: true dans Project Config.

8 agents support / audit

Isolation stricte

  • Chaque agent déclare ses reads/writes dans .claude/loader.yml. Le hook preflight rejette toute lecture hors patterns autorisés.
  • Le hook SubagentStop applique file-ownership.md §1 : refuse les Write hors matrice (par exemple : dev-frontend ne peut pas écrire dans {BackendName}/).
  • Concurrence cross-agent : file-ownership.md §4 impose un lock file pour {LibName}/ (cas DTOs partagés back+front).

3 — Le pipeline (4 phases)

Vue d'ensemble

Phase 1   /feat-generate Auth           → workspace/input/feats/1-Auth.md
Phase 1.5 /feat-deepen 1 [--quick]      → enrichit FEAT + constitution §7
Phase 2   /us-generate 1                → workspace/output/us/1-{m}-{Name}.md
Phase 2.6 /feat-validate 1              → validation déterministe (GO/WARN/NO-GO)
Phase 3   /arch-init                    → scaffolding projets + entities DB
Phase 3.5 /dev-plan 1                   → plans techniques sans coder
Phase 4a  dev-backend (parallèle)       → workspace/output/src/{BackendName}/
Phase 4b  QA API Gate                   → tests intégration HTTP (in-memory)
Phase 4c  dev-frontend (parallèle)      → workspace/output/src/{AppName}/
Phase 5   /qa-generate 1                → tests unitaires + coverage + 5 auditeurs

Tout cela est piloté par /sdd-full 1 qui orchestre les 4 phases en chaîne, avec gates manuels optionnels (--manual-gates=us,readiness,plan,code) résolus depuis la console web.

Backend-first gated workflow

Depuis 2026-05-07 (règle backend-first.md), /dev-run exécute dev-backend ALL US puis une API Gate (tests d'intégration HTTP in-memory) puis dev-frontend ALL US. Le frontend n'est généré que si l'API Gate passe en 🟢 ou 🟡.

4 — Base de données SQLite (v6.10 BREAKING)

Tous les outputs .json / .jsonl / .log de statistiques et télémétrie sont retirés du filesystem et centralisés dans une unique base SQLite en mode WAL.

Localisation & pragmas

workspace/output/db/console.db

PRAGMA journal_mode = WAL          # lectures concurrentes
PRAGMA synchronous  = NORMAL       # compromis durabilité/perf
PRAGMA busy_timeout = 5000         # tolère 5s d'attente sur lock
PRAGMA foreign_keys = ON

24 tables organisées en 4 familles

FamilleTablesÉcrit par
Méta schema_version init_console_db.py
Artefacts SDD (métadonnées uniquement) feats · us · plans · adrs helpers ensure_feat_row / ensure_us_row appelés par les writers
Runs & Telemetry runs · run_phases · gates · events · token_usage · context_budget · validation_reports · breaking_changes sdd_state.py · context_budget.py · gate_decide.py · hook record_token_usage.py
QA qa_coverage (+ qa_coverage_files) · qa_quality · qa_api_tests (+ qa_api_endpoints) · qa_a11y · qa_code_review · qa_security · qa_performance · qa_spec_compliance parse_coverage.py · quality_scan.py · agents auditor (via bridge ingest_agent_report.py)

Writers refactorés (v6.10)

Plus aucun fichier JSON de stats sur le FS. Tous les writers déterministes écrivent en DB directement :

  • sdd_state.py → tables runs / run_phases / events (remplace state/run-*.json + events.jsonl)
  • parse_coverage.py → qa_coverage + qa_coverage_files (remplace coverage.json)
  • quality_scan.py → qa_quality (remplace quality.json)
  • validate_fidelity.py → validation_reports (remplace fidelity-*.json)
  • context_budget.py → context_budget (remplace context-budget.jsonl)
  • record_token_usage.py (hook) → token_usage (remplace token-usage.jsonl)
  • gate_decide.py → table gates en dual-write avec workspace/console/status.json (la console UI reste compatible)

Bridge LLM → DB pour les agents auditor

Les 5 auditeurs LLM (a11y, code-review, security, performance, spec-compliance) ne peuvent pas écrire directement en DB. Ils émettent un JSON éphémère dans leur emplacement canonique, puis appellent ingest_agent_report.py --type X --feat N qui :

  1. parse le JSON
  2. INSERT dans la table qa_* correspondante
  3. supprime le JSON

Au final : aucun .json ne subsiste sur le FS. Le .md compagnon (lecture humaine) reste.

5 — Console web (Fastify + React)

Stack

  • Node.js ≥ 20 — runtime
  • Fastify 5 — serveur HTTP + SSE
  • @fastify/static — sert / (console SPA), /ui/ (mockups HTML avec leur CSS relatif)
  • React 18 via CDN unpkg (pas de build step)
  • @babel/standalone transforme app.jsx in-browser
  • @anthropic-ai/sdk — opt-in pour /api/explain (reformulation IA PO-friendly)
  • better-sqlite3 volontairement évité (native bindings) — la console requête console.db via subprocess Python (query_console_db.py). Zéro dépendance native.

UI

  • Theme : variables CSS --bg, --panel, --ink, --chart-1..7 commutées via :root[data-theme="dark"]. Toggle persisté en localStorage, suit prefers-color-scheme au premier load.
  • Charts : SVG natifs (donut, bar stacks, sparklines, gradient bars). Aucune librairie chart externe.
  • Dashboard sonar : ratings A→E avec couleurs dérivées (green / lime / amber / orange / red). Une métrique n'apparaît que si la table DB correspondante a des données (pas de placeholder vide).
  • Vue UX : carrousel <iframe src="/ui/X.html"> — les mockups chargent leur CSS relatif (design-system.css) sans duplication.
  • Documentation inline : /api/help/:id extrait le body et strip <style>/<script>/<link> + attrs style=/class= → injecté dans <article class="doc-content"> qui re-stylise via les variables CSS du site (cette page est rendue ainsi).

7 — Configuration en 3 couches (v6.7.1)

.claude/config.base.yml          # framework defaults (lecture seule)
~/.sdd/config.team.yml           # policy équipe / org (opt-in)
workspace/input/stack/stack.md   # ## Project Config (per-project)

Précédence : project > team > base. Sécurité : un project ne peut jamais relâcher la policy team sur SecurityFailOn / A11yFailOn / CoverageMin → ERROR [CONFIG_SECURITY_DOWNGRADE].

Snapshots de la policy team via /sdd-profile export strict-prod et /sdd-profile import dev-only. Stockage : ~/.sdd/profiles/{name}.yml.

8 — Serveur MCP (v6.9, opt-in)

Serveur Model Context Protocol qui expose 14 tools SDD Pro à des clients non-Claude-Code (Cursor, Windsurf, Cline, Claude Desktop, n8n, scripts CI). Stdlib pure (0 dépendance externe), protocole JSON-RPC 2.0 implémenté à la main.

Tools déterministes (7) — wrappers sdd_scripts/*.py

  • sdd_status · validate_readiness · feat_validate · set_us_status
  • validate_us_deps · compute_us_complexity · migrate_us_v1_to_v2

Tools LLM-driven (7) — subprocess claude CLI

  • claude_check · feat_generate · us_generate

9 — Taxonomie d'erreur unifiée

Tous les agents et scripts préfixent leurs blocs ERROR avec un code [CLASS] dans le CAUSE:. Permet à build_loop, hooks et dashboards de classer sans interprétation textuelle.

Format canonique 3 lignes

ERROR: {pipeline-step ou agent} — résumé
CAUSE: [{CLASS}] détail 1 ligne
FIX: action 1 ligne

Classes principales (~80 au total, voir error-classification.md)

  • Runtime : [NETWORK] · [AUTH] · [PERMISSION] · [NOT_FOUND] · [TIMEOUT] · [DISK] · [ENV_MISSING]
  • Pipeline : [STACK_MALFORMED] · [FEAT_REJECTED] · [READINESS_NO_GO] · [PLAN_STALE] · [INVALID_ARG]
  • Build (pilote build_loop) : [BUILD_CORRECTIBLE] (itère) · [BUILD_BLOCKING] (fail-fast)
  • QA : [QA_TEST_FAILED] · [QA_COVERAGE_GAP] · [QA_OUTPUT_INVALID]
  • Anti-derive : [DERIVE_VIOLATION] · [STACK_LIBRARY_MISSING] · [OPTIMIZATION_PROACTIVE]
  • A11Y (10) · REVIEW (11) · SEC (21) · PERF (16) · SPEC (6)
