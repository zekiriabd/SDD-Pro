# 🚀 Getting Started with SDD_Pro

**5 minutes to understand. 30 minutes to your first running project.**

This guide walks you from zero to a working full-stack app generated by SDD_Pro, with tests and quality reviews built in.

---

## What is SDD_Pro?

**Spec-Driven Development for Claude Code** : a framework that turns **functional specifications** into **production-ready code** through coordinated AI agents.

You write **what** the software should do (FEAT spec + optional HTML mockup). SDD_Pro orchestrates **12 AI agents** to figure out the **how** :

```
You write:               SDD_Pro produces:
─────────────────       ─────────────────────────────────
FEAT spec               • User Stories (atomic, traceable)
+ HTML mockup    ──→    • Backend code (services, endpoints, DTOs)
+ stack choice          • Frontend code (pages, components, theme)
                        • Unit tests (back + front)
                        • 5 quality review reports
                        • Coverage / lint / security / spec compliance
                        • Architectural decision records (ADRs)
```

The key innovation : **deterministic Python orchestration** (51 scripts, 0-cost) + **LLM agents** (12 specialized) + **5 cross-file reviewers** = pipeline that's auditable, idempotent, and resumable.

---

## How is it different from BMAD / Spec-Kit / etc.?

| Aspect | SDD_Pro | BMAD | Spec-Kit |
|---|---|---|---|
| Multi-agent | **12 specialized** | ~6 | 1 |
| Post-code reviewers | **5 distinct angles** | 1 | 0 |
| Anti-derive enforcement | strict (ownership matrix + STOP) | partial | ❌ |
| Stack catalog | `.libs.json` machine-readable + CVE + LTS check | ❌ | ❌ |
| Error taxonomy | **174 classes `[CLASS]`** cross-agent | ❌ | ❌ |
| Telemetry | SQLite (cost cap, audit trail, gates) | ❌ | ❌ |
| Determinism | 51 Python scripts (0 token) | ❌ | ❌ |
| Idempotence / resume | checkpoint mode | ❌ | ❌ |

**Differentiator** : SDD_Pro **industrializes quality** — it's the equivalent of **Sonar + Snyk + ADR governance** applied to LLM pipelines.

## 🏗 The big picture (5-minute tour)

### Phase model

```
┌─ Phase 1 ─ FEAT framing  ─────┐
│   /feat-generate Auth         │
│   ↓                            │
├─ Phase 1.5 (optional)         │
│   /feat-deepen Auth           │  ← elicitation (Pre-mortem, Red Team...)
│   ↓                            │
├─ Phase 2 ─ User Stories ──────┤
│   /us-generate 1              │
│   ↓                            │
├─ Phase 2.6 ─ Readiness gate   │
│   /feat-validate 1            │  ← deterministic gate, 0 token
│   ↓                            │
├─ Phase 2.7 (optional) ────────┤
│   /dev-plan 1                 │  ← human review of technical plans
│   ↓                            │
├─ Phase 3 ─ Architecture ──────┤
│   /arch-init                  │  ← scaffolding + DB
│   ↓                            │
├─ Phase 4 ─ Code (parallel) ───┤
│   dev-backend × N  ──→ API Gate ──→ dev-frontend × N
│   (build_loop ≤3 iter)         │
│   ↓                            │
├─ Phase 5 ─ QA ────────────────┤
│   /qa-generate 1              │  ← tests + coverage + quality
│   ↓                            │
├─ Phase 6 ─ Auditors (batch) ──┤
│   code-reviewer + security-reviewer + spec-compliance + arch-reviewer
│   ↓                            │
└─ Phase 7 ─ Verdict ───────────┘
    /sdd-review 1               ← consolidated 🟢/🟡/🔴
```

**One command runs all 7 phases :** `/sdd-full 1`

## ⚡ The 30-minute tutorial

### Prerequisites

- **Python ≥ 3.10** (`python --version`)
- **Node.js ≥ 20** (for the optional web console)
- **Git**
- **An Anthropic API key** (for Claude Code) — `export ANTHROPIC_API_KEY=sk-ant-...`
- **Claude Code** installed (`https://claude.com/claude-code`)
- One stack-specific runtime depending on combo (e.g. .NET 10, Node 22, etc.)

---

### Step 1 — Get the framework (2 min)

Use this repo as a **template** (recommended) :

1. Copy the repo to a new private location (Azure DevOps clone, fork, etc.)
2. Clone it locally :

```bash
git clone <your-repo-url> my-first-sdd-app
cd my-first-sdd-app
```

---

### Step 2 — Bootstrap (5 min)

Run the interactive bootstrap. You'll be asked for the project name + 3-4 stack choices.

```bash
# macOS / Linux
python3 bootstrap.py

# Windows PowerShell
.\bootstrap.ps1
```

**Recommended for first try** : combo **C1** = .NET Minimal API + React + shadcn + Azure AD + xUnit.

```bash
python bootstrap.py --combo c1
```

What happens :
- ✅ `workspace/input/stack/stack.md` generated (43 Project Config keys)
- ✅ `workspace/output/.sys/` structure created
- ✅ Python deps installed (`pip install -e .claude/python[dev]`)
- ✅ Console deps optionally installed (`workspace/console/`)
- ✅ Smoke check passes

After this, your repo is **initialized** but contains no FEAT yet.

---

### Step 3 — Edit secrets (1 min)

Open `workspace/input/stack/stack.md` (already opened by bootstrap on most editors) and replace placeholders :

```yaml
## Active Database
Type: PostgreSql
Server: localhost
Database: my_first_app
User: postgres
Password: my_dev_password   # ← edit
Port: 5432

## Active Auth Specs
AzureAD.TenantId: 00000000-0000-0000-0000-000000000000  # ← edit (or use auth-local)
AzureAD.ClientId: 11111111-1111-1111-1111-111111111111  # ← edit
```

> 🔒 `stack.md` is **gitignored** — secrets stay local. The arch agent will propagate values to `appsettings.json` / `application.yml` etc. at scaffold time.

---

### Step 4 — Your first FEAT (5 min)

In Claude Code, run :

```
/feat-generate Auth
```

You'll be asked 3-6 questions :
- What does Auth do?
- Who are the actors (PO, end users, admins, etc.)?
- Main flows?
- Business rules?
- Acceptance criteria?

The agent produces `workspace/input/feats/1-Auth.md` with structured sections : `## Functional Needs`, `## Functional Deliverables`, `## Business Rules`, `## Acceptance Criteria`, each with stable IDs `SFD-N`, `FD-N`, `BR-N`, `AC-N`.

**Optional** : drop a mockup HTML at `workspace/input/ui/1-2-Login.html` for visual fidelity.

### Step 6 — Inspect the result (2 min)

```bash
# Check the status
/sdd-status 1

# Look at the generated code
ls workspace/output/src/MyAppBack/
ls workspace/output/src/MyAppFront/

# See the consolidated review
cat workspace/output/qa/feat-1/review.md
```

---

### Step 7 — Run it (optional, 2 min)

```bash
/sdd-serve            # launches backend + frontend + console in background
```

Open in browser :
- **Web app** : http://localhost:5173 (or whatever port your stack uses)
- **API** : http://localhost:8080 (Swagger UI exposed)
- **SDD Console** : http://localhost:4000 (telemetry dashboard)

When done :
```bash
/sdd-kill-server
```

## ❓ Common questions during first use

### "Will SDD_Pro modify my existing code?"

No. SDD_Pro writes **only under `workspace/output/src/`** (its own sandbox). Your existing repo source code is never touched.

### "What if my FEAT is too vague?"

Use `/feat-deepen 1` (the `elicitor` agent) to enrich it with 5 angles (Pre-mortem, Red Team, RACI, Failure Modes, Inversion). Run this BEFORE `/sdd-full` for critical features.

### "What if a phase fails halfway?"

```bash
/sdd-full 1 --resume
```

The resume logic uses `sdd_state.py resume-target` to skip past completed phases. Each phase is idempotent.

### "How do I review the agents' work step by step?"

```bash
/sdd-full 1 --plan --manual-gates=us,plan,code
```

This pauses after each major phase, letting you inspect + approve in the web console (or by editing `workspace/console/status.json`).

### "Is my data secure?"

- `workspace/input/stack/stack.md` (secrets) is gitignored.
- `workspace/console/.certs/` (HTTPS keys) is gitignored.
- No telemetry leaves your machine (SQLite is local).
- Anthropic API calls follow Claude Code's policy.

### "Can I use SDD_Pro for production?"

Yes for **combos C1 and C2** (🟢 validated end-to-end). Other combos (C3-C5) are bench-validated runtime but the full `/sdd-full` pipeline has some manual scaffolding gaps. See [validated-combos.md](validated-combos.md).

## 🎓 Mental model in 3 ideas

1. **You write specs, agents write code.** The contract is the FEAT spec (with traceability IDs). The output is code that compiles, tests pass, and 5 reviewers approve.

2. **Quality is industrialized, not aspirational.** Every output is checked by deterministic Python scripts (coverage parsing, quality scan, OWASP regex) AND by LLM agents (cross-file review). Hard-blocking errors cannot be bypassed without a tracked decision.

3. **The framework refuses to drift.** Anti-derive rules (`build-and-loop §3`, ownership matrix, plan staleness checks) STOP the pipeline rather than guessing. The error message tells you exactly what to fix.

This is the **Sonar + Snyk + ADR governance** of LLM-assisted development. Welcome aboard.
